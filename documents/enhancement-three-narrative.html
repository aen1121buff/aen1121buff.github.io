Enhancement Three: Databases Artifact Description The artifact I selected for this database enhancement is my CS 340 Grazioso Salvare dashboard project. This project was originally created in CS 340: Advanced Programming Concepts. It uses Python, MongoDB, Dash, Pandas, Plotly, and Dash Leaflet to display animal shelter records from the Austin Animal Center dataset. The dashboard was designed to help Grazioso Salvare identify dogs that may be good candidates for different types of rescue training. The original version included a reusable MongoDB CRUD module, a database connection, rescue type filtering, an interactive data table, a chart, and a map. This Milestone Four enhancement builds directly on my latest approved CS 499 artifact files from Milestone Three. In Milestone Three, I enhanced the dashboard with rescue scoring and ranked sorting. For Milestone Four, I kept those improvements and added a database focused enhancement. The main changes were made to animal_shelter.py and dashboard_enhanced.py, with a new database_tests.py file added to help verify the database connection, indexes, counts, and sample filtered records. Justification for Inclusion in the ePortfolio I selected this artifact for my ePortfolio because it shows how database work supports a complete software solution. The dashboard depends on MongoDB to store animal shelter records and retrieve filtered data for users. This makes the project a strong database artifact because it demonstrates more than a simple query. It shows database connection handling, CRUD operations, filtering, validation, indexing, and how data moves from MongoDB into an interactive dashboard. The artifact showcases my database skills because the enhanced version improves the reusable MongoDB access layer. The AnimalShelter class now validates database queries, projections, sorting input, update values, and delete queries. It also validates the MongoDB connection with a ping before dashboard queries run. I added index creation for the fields used most often by the dashboard filters, including animal_type, breed, sex_upon_outcome, age_upon_outcome_in_weeks, and outcome_type. I also added a compound index for the rescue filter fields so the dashboard can retrieve common rescue searches more efficiently. The artifact was improved because the dashboard no longer reads unnecessary fields from the database. The enhanced code uses a controlled projection so only the fields needed for the table, chart, map, and scoring logic are returned. This is a better database design choice because it limits data exposure and makes dashboard reads more efficient. I also improved delete behavior so the database class deletes one record by default unless multiple record deletion is specifically allowed. This reduces the risk of accidental bulk deletion. The dashboard was updated to use the improved database methods, display database record counts, show index status, and handle database errors more clearly. Reflection on the Enhancement Process While enhancing this artifact, I learned that database quality affects the whole application. The dashboard can only be useful if the database layer is reliable, safe, and easy to reuse. In earlier versions, the dashboard could filter records and display data, but the database access layer needed more protection and structure. Adding validation, index support, controlled projections, and safer helper methods made the project feel more professional and easier to maintain. One challenge I faced was deciding how far to change the database layer without breaking the existing dashboard. I wanted to improve the database code while still keeping the same purpose of the CS 340 project. I handled this by keeping the same AnimalShelter class structure and adding stronger helper methods around it. Another challenge was balancing performance and simplicity. I added index creation and dashboard read limits to improve performance, but I kept the logic clear so another developer could understand it quickly. This enhancement helped me understand that database work is not only about storing and retrieving records. It also involves thinking about validation, safety, performance, and maintainability. The final artifact is stronger because the database layer now supports the dashboard in a cleaner and safer way. It also shows my ability to improve an existing project without removing the previous algorithm and dashboard enhancements.
